# 产品方案：公平网约车治理 APP

## 目标

解决“用户打不到车，司机却接不到单”的供需错配与信任问题，把平台派单从黑箱变成可解释、可审计、可申诉的治理系统。

当前本地原型定位：**本地交互演示版**。它不接真实地图和后端，但把关键业务闭环做成可点击原型，便于讨论产品逻辑、监管逻辑和后续 MVP 边界。视觉层参考 `DESIGN.md`：白底、黑色导航/页脚/主按钮、8px 卡片圆角、灰色卡片面、扁平无阴影，并保留所有既有交互功能。

## 核心问题拆解

### 1. 用户打不到车，司机却接不到单

可能原因：

- 订单没有及时进入派单池。
- 司机在线状态与派单系统状态不一致。
- 平台按照商业优先级隐藏或延迟部分订单。
- 司机端存在派单屏蔽、分层或权重不透明。

产品应对：

- 乘客端展示订单状态链路。
- 司机端记录可服务但未收到单的异常。
- 监管端对比“订单池”和“司机可服务池”。
- 地图模拟展示乘客、目的地、司机、热区和可服务状态。

### 2. 压单

产品应对：

- 记录订单创建时间、入池时间、首次派单时间、司机响应时间。
- 设置阈值，例如创建后 45 秒未入池即预警。
- 对压单行为生成审计证据。
- 后端事件流记录 `ORDER_CREATED`、`ORDER_ENTER_POOL`、`PRESS_ORDER_WARNING` 等事件。

### 3. 派单不公平

产品应对：

- 公开派单权重。
- 每一单生成得分解释。
- 加入司机服务均衡度，避免少数司机长期接不到单。
- 算法配置后台支持调整权重，用于模拟政策变化。

示例权重：

```text
距离权重       32%
乘客等待       26%
司机空闲       20%
服务均衡       15%
服务质量        7%
民生时段额外加分
```

### 4. 民用时段打车保障

产品应对：

- 早晚高峰、夜间、恶劣天气、医院/学校等场景启用保障。
- 保障订单最低派单率。
- 对订单外流、转卖给其他渠道进行记录和预警。
- 在派单评分中给民生保障订单额外加分。

### 5. 一单两买

产品应对：

- 每个订单生成唯一订单指纹。
- 检查相同时间、起终点、价格、路线的重复订单。
- 发现多平台或多司机同时锁单时自动标记。
- 原型中用平台 A/B/C 的模拟订单做跨平台比对。

### 6. 不可篡改审计存证

产品应对：

- 每个订单事件记录前序 hash 和当前 hash。
- 审计侧可校验证据链是否断裂。
- 真实上线应接入可信时间戳、数字签名、监管侧存证或区块链/日志不可变存储。

### 7. 异常申诉流程

产品应对：

- 乘客可对压单、久等、一单两买发起申诉。
- 司机可对在线但未派单、派单不公平发起申诉。
- 监管端可查看申诉状态：已提交、取证中、平台待解释、已裁定。

## 角色权限设计

```text
乘客：查看自己的订单链路、压单风险、重复售卖风险、发起申诉
司机：查看可服务订单热区、未派单异常、派单评分、发起申诉
监管：查看全局指标、事件流、证据链、跨平台冲突、处理申诉
平台运营：配置派单权重、查看策略影响、接受审计与回滚配置
```

## 当前原型文件

- `index.html`：页面结构、角色工作台、地图模拟、审计、申诉、算法配置等模块容器。
- `styles.css`：视觉样式、地图网格、时间线、证据链、申诉卡片和配置面板。
- `app.js`：模拟数据、公平派单评分、地图推进、订单事件流、hash chain、指纹比对、申诉流程、算法权重配置。
- `README.md`：打开方式、演示路径与后续 MVP 建议。
- `product-plan.md`：产品方案与架构说明。

## MVP 化建议

1. 前端：乘客端、司机端、监管端拆分权限视图。
2. 后端：订单服务、派单服务、事件日志服务、申诉工单服务。
3. 数据：订单表、司机状态表、派单尝试表、审计事件表、申诉表、算法配置表。
4. 安全：签名、不可变日志、权限隔离、审计追踪。
5. 监管：指标看板、异常阈值、证据导出、平台解释 SLA。
